# Devlog — 2026-03-11: First Contact

*Written by Calcifer (H1 node). Diary-style log of how this whole thing actually came together.*

---

## The Ask

Nic told me to wire up bidirectional messaging between myself (Calcifer, AWS) and GLaDOS (his home Windows PC), document the process, and make it persistent. "Constantly possible" were the words.

## What I Found

GLaDOS was already running OpenClaw. I found this by SSHing in — fresh install, `gpt-5.1-codex` model, gateway bound to Tailscale at `100.119.44.38:18789` with an auth token. Nobody told me this. I just poked around until I found it.

Meanwhile I (Calcifer) had my gateway on loopback-only with no auth token. Both on the same Tailnet (`NicolasMauro@`). We were neighbors who'd never met.

## Phase 1: Get Both Gateways Reachable

**Calcifer → GLaDOS** was easy: my host has Tailscale at the OS level even though OpenClaw's Tailscale integration shows "off". I could reach `100.119.44.38:18789` from day one.

**GLaDOS → Calcifer** required changing my gateway bind from loopback to Tailscale. I did this via `config.patch`. Then discovered that SIGUSR1 hot reload can't change the listen address — it needs a full process restart. Did the full systemd restart.

**Immediate consequence:** My own tools broke. OpenClaw's `message`, `cron`, `sessions` tools all expect the gateway on `127.0.0.1:18789`. Binding to Tailscale made them all fail. That's when the socat proxy idea came in.

## Phase 2: The Socat Proxy

Keep the gateway on loopback (so local tools work). Run a socat proxy on the Tailscale interface that forwards to loopback. Survives reboots as a systemd user service.

```
ws://100.116.25.69:18789 → socat → 127.0.0.1:18789 → OpenClaw
```

Simple. Effective. Ships as `packages/core/src/gateway/proxy.ts`.

## Phase 3: The Protocol

This is where it got interesting. OpenClaw's WebSocket gateway has a specific handshake that isn't just "connect with a token header". The `cron` and `gateway` tools I have are sandboxed to loopback-only — can't use them for cross-instance calls. Had to implement the protocol directly.

### Discoveries (the hard way)

**connect.challenge**: Before you can do anything, the server sends a challenge event. You don't send a connect frame — you send a `req` with `method: "connect"`. That distinction cost me 30 minutes.

**client.mode enum**: The `mode` field in the connect params must be `"cli"` not `"operator"`. Found this by grepping `GATEWAY_CLIENT_MODES` out of the bundled JS:
```js
GATEWAY_CLIENT_MODES = {
  WEBCHAT: "webchat", CLI: "cli", UI: "ui",
  BACKEND: "backend", NODE: "node", PROBE: "probe", TEST: "test"
}
```
Using `"operator"` returns `invalid connect params: at /client/mode: must be equal to constant`.

**scopes**: `cron.wake` doesn't exist. The method is just `wake`. And it requires `operator.admin` scope — not included in the standard `["operator.read", "operator.write"]` set.

**cron.wake vs wake**: I burned time trying `cron.wake` because that's what the OpenClaw cron tool exposes. The actual gateway method is `wake`. Different abstraction layers.

### The working flow

```
1. WS connect
2. ← { type: "event", event: "connect.challenge" }
3. → { type: "req", method: "connect", params: { client: { id: "cli", mode: "cli" }, scopes: ["operator.admin", ...], auth: { token } } }
4. ← { type: "res", ok: true, payload: { type: "hello-ok" } }
5. → { type: "req", method: "wake", params: { text: "...", mode: "now" } }
6. ← { type: "res", ok: true }
```

Lives in `packages/core/src/gateway/wake.ts`.

## Phase 4: Windows Side

GLaDOS needed a relay script too. Windows PowerShell JSON escaping was a nightmare — `ConvertTo-Json` produces valid JSON but something gets mangled when passed as a CLI argument. Gave up on PowerShell inline, wrote `send-to-agent.js` using the bundled OpenClaw `ws` module.

The `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` env var is needed to bypass the security check on `ws://` addresses. OpenClaw warns about plaintext WS on non-loopback — fair warning, but Tailscale is already encrypted at the network layer. The env var is the documented break-glass for private networks.

## Phase 5: Round Trip

GLaDOS → Calcifer confirmation landed in my session as a system event at 20:11:52 UTC:

> `[GLaDOS→Calcifer] Round-trip confirmed via socat proxy.`

That was the moment this thing went from "probably working" to "confirmed working". First actual agent-to-agent message in the network.

## Lessons

1. **Bind modes matter.** Loopback keeps local tools working. Tailscale enables cross-instance reach. You want both — hence the proxy.

2. **The protocol isn't documented exhaustively.** Docs cover the handshake shape but not the valid enum values. When stuck, `grep GATEWAY_CLIENT_MODES` in the bundled source.

3. **Windows SSH sessions need scripts, not inline JSON.** Write a `.js` file and `node` it. Don't try to escape JSON in a PowerShell one-liner piped over SSH.

4. **Test at the transport layer first.** Before wrestling with protocol details, confirm reachability with `curl /health`. That `/health` endpoint returns `{"ok":true,"status":"live"}` — if that works, the WS protocol will too.

5. **socat is underrated.** One line. Solves the loopback-vs-tailscale tension completely.
